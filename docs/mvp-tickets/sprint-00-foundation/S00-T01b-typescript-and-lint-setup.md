# S00-T01b: TypeScript and Lint Setup

Sprint: 00 - Foundation

## Objective

Convert the existing codebase to strict TypeScript and add ESLint so that type
checking and linting become part of the standard verification gate every later
ticket must pass. This tightens the automated feedback loop for AI-driven
development, where type and lint errors are the primary signal that catches
mistakes before human review.

## Exact Scope

- Add `tsconfig.json` with `strict: true`.
- Add an ESLint flat config wired to TypeScript and Next.js.
- Convert all existing `src/**` source and test modules from `.mjs`/`.jsx` to
  `.ts`/`.tsx` with correct types, preserving current behavior exactly.
- Add `typecheck` and `lint` scripts and fold them into `bun run check`.
- Do not change any runtime behavior, API shapes, or test assertions.
- Do not add new features, routes, or dependencies beyond the TypeScript and
  ESLint toolchain.
- Leave `next.config.mjs` and `eslint.config.mjs` as `.mjs` (config files).

## Backend changes

- Add dev dependencies: `typescript` (pin to the 5.x line, e.g. `~5.9.0`),
  `@types/node`, `@types/react`, `@types/react-dom`, `eslint` (pin to the 9.x
  line, e.g. `~9.39.0`), `eslint-config-next`, `typescript-eslint`.
  - Pin `typescript` to 5.x: `latest` resolves to the 7.x native compiler,
    whose changed internal API crashes `typescript-eslint@8` during lint.
  - Use the official `eslint-config-next/core-web-vitals` preset rather than
    wiring `@next/eslint-plugin-next` and `eslint-plugin-react-hooks` by hand.
    This entry point (`import nextConfig from "eslint-config-next/core-web-vitals"`)
    is a ready flat-config array — the base `eslint-config-next` rules
    (`@next/next/*`, `react/*`, `react-hooks/*`, `jsx-a11y/*`, `import/*`) plus
    `@next/eslint-plugin-next`'s `core-web-vitals` overrides layered on top —
    and stays current as Next.js evolves, rather than us re-selecting a subset
    of rule families by hand and drifting from Next's own recommendations.
    Confirmed empirically: the `core-web-vitals` layer changes exactly two
    rules from `warn` to `error` versus the base preset —
    `@next/next/no-html-link-for-pages` (catches `<a href>` used for internal
    navigation instead of `<Link>`) and `@next/next/no-sync-scripts`
    (catches blocking `<script>` tags). Everything else keeps its base
    severity (e.g. `no-img-element` stays `warn`). Verified the existing
    same-page `<a href="#section">` anchors in `page.tsx` do not false-positive
    on `no-html-link-for-pages` — that rule targets links to other app routes,
    not in-page anchors. The preset does not include `typescript-eslint`'s
    rule set (only parser/plugin registration for `.ts`/`.tsx`), so
    `typescript-eslint`'s `configs.recommended` is still layered on top for
    `@typescript-eslint/*` rules.
  - Pin `eslint` to 9.x: `latest` resolves to ESLint 10, but
    `eslint-plugin-react@7.37.5` (pulled in transitively by
    `eslint-config-next`) peer-depends on `eslint@^9.7` and crashes on
    ESLint 10's changed rule-context API (`context.getFilename` removed).
  - `@next/next/no-img-element` and `jsx-a11y/alt-text` still ship as `warn`
    even under `core-web-vitals`, and `eslint .` does not fail the process on
    warnings alone — that is the preset's own intentional default, not a gap
    to patch here.
- `tsconfig.json` compiler options must include at least:
  - `"strict": true`
  - `"noEmit": true`
  - `"allowImportingTsExtensions": true` (Node's test runner resolves the real
    file on disk, so relative imports keep explicit `.ts` extensions)
  - `"module": "esnext"`, `"moduleResolution": "bundler"`
  - `"isolatedModules": true`, `"verbatimModuleSyntax": true` (these keep every
    file independently transformable, which is what Node's native type
    stripping requires; use `import type` for type-only imports)
  - `"jsx": "react-jsx"`, `"target": "ES2022"`, `"skipLibCheck": true`
    (Next's build rewrites `jsx` to `react-jsx` automatically, so set it there
    from the start rather than `preserve`)
  - `"plugins": [{ "name": "next" }]`
- Keep explicit file extensions in relative imports, now `.ts`/`.tsx`
  (e.g. `import { loadLimits } from "./limits.ts";`).
- Update `package.json` scripts:
  - `"test": "node --test src/test/*.test.ts"`
  - `"typecheck": "next typegen && tsc --noEmit"` (regenerate Next.js route,
    page, layout, and route-handler types before every standalone type check so
    the result cannot depend on stale `.next` output)
  - `"lint": "eslint ."`
  - `"check": "bun run lint && bun run typecheck && bun run test && bun run build"`
- Tests run on Node's native TypeScript type stripping (Node 24+), so no
  `ts-node` or `tsx` runtime dependency is required. Avoid non-erasable syntax
  (TypeScript `enum`, `namespace`, parameter properties, and decorators) so the
  test runner can strip types without a transform step.

## Acceptance criteria

- `bun run lint` passes with zero errors.
- `bun run typecheck` regenerates Next.js types and passes with zero errors
  under `strict: true`.
- `bun run test` passes all previously passing tests, unchanged in behavior.
- `bun run build` passes.
- `bun run check` runs lint, typecheck, test, and build in that order and passes.
- No `.mjs` or `.jsx` files remain under `src/` (config files at the repo root
  may stay `.mjs`).
- No runtime behavior, API response shape, or test assertion changed.
